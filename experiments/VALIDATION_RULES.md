# Validation Rules — The Fitness Function

This file defines what counts as "better" or "worse" for any experiment.
The agent uses these rules to filter results before proposing variations.

## Priority Order (LOCKED — do not reorder)

Kevin's stated goals in priority order:
1. MAXIMIZE ACCURACY (win rate) — most important
2. MINIMIZE RISK (drawdown, loss size)
3. MAXIMIZE PROFIT (total P&L)

When ranking experiments, sort by P&L within configs that pass all gates.
When two configs are close, prefer the one with higher win rate.
When win rates are close, prefer the one with lower drawdown.

## HARD GATES — Auto-reject if any of these fail

An experiment is REJECTED (cannot be a "winner") if ANY of these are true:

- **Insufficient trades**: trades < 30
  → Not statistically meaningful. Cannot evaluate.

- **Win rate collapse**: win_rate < (baseline_win_rate - 3.0%)
  → Accuracy is priority 1. A 3% absolute WR drop is unacceptable regardless of P&L.

- **Drawdown doubled**: max_drawdown > (baseline_max_drawdown × 2.0)
  → Risk priority 2. Configs that double the risk are rejected.

- **R:R floor breached**: rr_ratio < 1.5
  → Even a high-WR config with poor R:R is a fragile edge.

- **No exits hit TP**: 100% of exits are stop_loss or breakeven_stop
  → Suggests the strategy never reaches its targets. Setup is broken.

## SOFT SIGNALS — Note but don't auto-reject

These reduce confidence in a winner but don't disqualify:

- **Trade count drop > 50%**: from baseline. Means the filter is too tight.
  Still a winner if other metrics improve, but flag as "narrow."

- **Win rate at exactly baseline**: no improvement on priority 1. Only a winner
  if P&L improvement is >20%.

- **Year-by-year inconsistency**: if a strategy is profitable overall but loses
  3+ of 5 years, it's regime-dependent. Flag as "regime-fragile."

## WINNER DEFINITION

An experiment is a WINNER if ALL of these are true:

1. Passes ALL hard gates above
2. Total P&L exceeds baseline by ≥5%
3. Either win rate ≥ baseline win rate, OR P&L improvement ≥20%

## CONVERGENCE DEFINITION

The loop is CONVERGED when:

- Last 3 consecutive generations produced 0 winners that improve current best
  P&L by ≥5%, OR
- Last 10 experiments have all been REJECTED, OR
- Current best config has been the best for 5+ generations

When CONVERGED, the agent writes to STATUS.md and stops proposing experiments.

## Baseline Reference

The baseline is `experiments/results/baseline.json`. It is generated by the
first run of `run_experiments.py` using the production config (no overrides).

If baseline.json doesn't exist, NO experiment can be a winner — all proposals
must be exploratory and the loop runs the baseline first.

## Exit Reason Quality

Beyond raw P&L, exit reason mix indicates strategy health:

- **Healthy**: 30%+ exits at tp1 or tp2 (winners actually reaching targets)
- **Marginal**: 15-30% TP exits, rest breakeven/stop
- **Broken**: <10% TP exits, mostly breakeven_stop (winners cut early)

If an experiment has <10% TP exits AND lots of breakeven_stop exits, flag this
as `broken_exit_pattern` even if P&L is positive. Likely indicates the breakeven
guard is firing too early.
